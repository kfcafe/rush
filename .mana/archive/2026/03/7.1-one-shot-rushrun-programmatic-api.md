Add a one-shot programmatic API to rush's library crate so external Rust consumers (like Tower's imp) can execute shell commands without managing lexer/parser/executor lifecycle.

## What to build

A `rush::run()` function and supporting types, exposed from `src/lib.rs`:

```rust
/// Options for programmatic command execution
pub struct RunOptions {
    /// Working directory (defaults to current dir)
    pub cwd: Option<PathBuf>,
    /// Environment variables (defaults to current env)
    pub env: Option<HashMap<String, String>>,
    /// Timeout in seconds (None = no timeout)
    pub timeout: Option<u64>,
    /// Request JSON output from built-in commands
    pub json_output: bool,
    /// Max output bytes (truncates if exceeded, None = no limit)
    pub max_output_bytes: Option<usize>,
}

/// Structured result from command execution
pub struct RunResult {
    pub exit_code: i32,
    pub stdout: String,
    pub stderr: String,
    /// true if output was truncated due to max_output_bytes
    pub truncated: bool,
    /// true if command was killed due to timeout
    pub timed_out: bool,
}

/// Execute a shell command string and return structured output.
/// Uses Executor::new_embedded() internally — no terminal interaction.
pub fn run(command: &str, options: &RunOptions) -> Result<RunResult>
```

## Implementation approach

- Create new module `src/run_api.rs`
- Inside `run()`: create Lexer, lex the command, parse with Parser, create `Executor::new_embedded()`, execute
- Apply cwd by calling `runtime.set_cwd()` before execution (check how Executor exposes runtime — may need a `pub fn runtime_mut(&mut self)` accessor, or set cwd via the environment/constructor)
- Apply env by setting env vars on the runtime
- Apply timeout: spawn execution in a thread, join with timeout, kill if exceeded
- Map `ExecutionResult` to `RunResult`
- If `max_output_bytes` is set, truncate stdout/stderr
- Re-export `run`, `RunOptions`, `RunResult` from `src/lib.rs`
