## Task

Create a provider-agnostic LLM client that supports Ollama (local), OpenAI, and Anthropic. This replaces the current Pi-specific client in `src/daemon/pi_client.rs`.

## Design

```rust
// src/ai/mod.rs
pub mod client;
pub mod providers;
pub mod config;

// src/ai/client.rs
pub struct LlmClient {
    provider: Box<dyn LlmProvider>,
    config: LlmConfig,
}

pub trait LlmProvider: Send {
    fn chat(&self, messages: &[Message], tools: Option<&[Tool]>) -> Result<Response>;
    fn name(&self) -> &str;
    fn supports_tools(&self) -> bool;
}

pub struct Message {
    pub role: Role,       // System, User, Assistant, Tool
    pub content: String,
}

pub struct Tool {
    pub name: String,
    pub description: String,
    pub parameters: serde_json::Value,  // JSON Schema
}

pub enum Response {
    Text(String),
    ToolCall { name: String, arguments: serde_json::Value },
}
```

## Providers

### Ollama (src/ai/providers/ollama.rs)
- HTTP API at `localhost:11434`
- Endpoint: `POST /api/chat`
- Supports tool calling with compatible models (qwen2.5-coder, llama3.1+)
- No API key needed
- Use `ureq` or raw `std::net::TcpStream` — do NOT use reqwest (too heavy)

### OpenAI (src/ai/providers/openai.rs)
- Standard chat completions API
- Requires OPENAI_API_KEY env var
- Tool calling via function_call

### Anthropic (src/ai/providers/anthropic.rs)
- Messages API
- Requires ANTHROPIC_API_KEY env var
- Tool use via tool_use content blocks

## Config (src/ai/config.rs)
```rust
pub struct LlmConfig {
    pub provider: ProviderType,  // Ollama, OpenAI, Anthropic
    pub model: String,
    pub api_key: Option<String>,
    pub base_url: Option<String>,
}
```

Stored in `~/.rush/ai.toml`:
```toml
provider = "ollama"
model = "qwen2.5-coder:7b"
```

## Key constraint
- Do NOT use reqwest. Use `ureq` (sync, minimal deps) or raw TCP+TLS for HTTP.
- The client must be sync (rush is not async)
- Tool calling support is required for the agent (unit 16.2.3)
