## Task

Add support for `[[ expression ]]` (bash extended test). This is not POSIX but is 
extremely common in scripts and heavily used by AI agents.

**Current error:** `rush: Invalid token at position 0: '[[ -f /etc/hosts '`

## Implementation

### Step 1: Lexer (src/lexer/mod.rs)
Add tokens for `[[` and `]]`:
```rust
#[token("[[")]
DoubleLeftBracket,

#[token("]]")]
DoubleRightBracket,
```
Must come BEFORE `LeftBracket` for priority.

### Step 2: Parser (src/parser/mod.rs)
When `DoubleLeftBracket` is encountered:
- Parse it as a command named `[[ `
- Collect arguments until `]]` is found
- Return a Command with name `[[` and the arguments

The simplest approach: treat `[[ expr ]]` as a command (like `[`) and let the 
test builtin handle evaluation. In `parse_command`:
```rust
Some(Token::DoubleLeftBracket) => "[[".to_string(),
```
And in the argument loop, stop when `DoubleRightBracket` is encountered.

### Step 3: Builtins (src/builtins/test.rs)
The test builtin should handle invocation as `[[` in addition to `[` and `test`.
`[[` supports additional operators vs `[`:
- `=~` for regex matching
- `==` and `!=` with glob patterns
- `&&` and `||` inside the expression
- No word splitting on variables (safer)

For initial implementation, just delegate to the same test logic as `[`.

## Files
- src/lexer/mod.rs (modify — add DoubleLeftBracket, DoubleRightBracket tokens)
- src/parser/mod.rs (modify — handle [[ as command name, ]] as terminator)
- src/builtins/test.rs (modify — handle [[ invocation)

## Acceptance
- `rush -c '[[ -f /etc/hosts ]] && echo yes'` outputs "yes"
- `rush -c '[[ -z "" ]] && echo empty'` outputs "empty"
- `rush -c '[[ "hello" == "hello" ]] && echo match'` outputs "match"
- `rush -c 'if [[ -d /tmp ]]; then echo dir; fi'` outputs "dir"
